# Резервное копирование и восстановление PostgreSQL

## Цель

Защитить рабочие данные Telegram-бота «ЭРА» от удаления, повреждения базы, ошибочной миграции и сбоя инфраструктуры.

## Текущая инфраструктура

База описана в `render.yaml` как Render PostgreSQL `era-postgres` на тарифе `free`. Репозиторий не должен содержать строку подключения или другие секреты.

## Политика

| Параметр | Значение |
|---|---|
| Частота | ежедневно в 01:17 UTC |
| Формат | PostgreSQL custom dump |
| Проверка | автоматическое восстановление в изолированную PostgreSQL |
| Контроль целостности | SHA-256 |
| Хранение | 30 дней в GitHub Actions artifacts |
| RPO | не более 24 часов |
| Целевой RTO | до 2 часов |

## Первичная настройка

В GitHub необходимо создать Actions secret:

`BACKUP_DATABASE_URL`

Значение — внешняя строка подключения Render PostgreSQL с пользователем, имеющим права только на чтение всех таблиц и последовательностей, если это поддерживается текущей схемой. Не использовать токен бота и не помещать URL в код, issue, PR или логи.

После добавления секрета вручную запустить workflow `Database backup`. Успешным считается только запуск, в котором одновременно выполнены:

1. `pg_dump` завершён без ошибки;
2. создан непустой dump;
3. совпала контрольная сумма;
4. dump восстановлен в отдельную временную PostgreSQL;
5. обнаружена обязательная таблица `users`;
6. workflow завершился зелёным статусом.

## Ежедневная работа

Workflow `.github/workflows/database-backup.yml` запускается автоматически. Каждый запуск создаёт отдельный artifact с dump и manifest-файлом.

Проверять минимум раз в неделю:

- последний запуск успешен;
- последняя копия не старше 24 часов;
- размер dump не равен нулю;
- restore verification завершён успешно.

## Восстановление

Восстановление сначала проводится только в новой тестовой базе.

```bash
export RESTORE_DATABASE_URL='postgresql://.../era_restore_test'
export BACKUP_FILE='backups/era-postgres-YYYYMMDDTHHMMSSZ.dump'
export BACKUP_SHA256='значение из manifest'
./scripts/verify_database_restore.sh
```

После восстановления проверить:

- количество пользователей;
- количество транзакций баллов;
- количество мероприятий и регистраций;
- наличие связей между пользователями, департаментами и направлениями;
- отсутствие отрицательных или необъяснимых балансов;
- текущую версию Alembic;
- запуск бота на тестовой строке подключения.

Переключать production на восстановленную базу можно только после фиксации причины инцидента, сохранения аварийной копии повреждённой базы и подтверждения владельца проекта.

## Откат

1. Остановить новые записи в повреждённую базу.
2. Создать аварийный dump текущего состояния.
3. Развернуть последнюю проверенную копию в новой базе.
4. Выполнить миграции только после сверки версии схемы.
5. Проверить ключевые таблицы и баланс контрольных пользователей.
6. Обновить `DATABASE_URL` в Render.
7. Перезапустить сервис и проверить `/health`.
8. Сохранить журнал инцидента.

## Ограничения

- GitHub workflow не сможет работать, пока не задан `BACKUP_DATABASE_URL`.
- Репозиторий не может самостоятельно создать или прочитать Render secret.
- Artifact не заменяет независимую копию в другом облаке. Следующий уровень защиты — S3-совместимое хранилище с versioning и lifecycle policy.
- Не хранить dump в Git, публичных ссылках или обычных сообщениях.

## Ежеквартальная проверка

Раз в три месяца выполнить полное учебное восстановление и зафиксировать:

- дату копии;
- время восстановления;
- результат проверки целостности;
- найденные отклонения;
- фактический RPO и RTO.
